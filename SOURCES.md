# Research Sources & Sample Data (`SOURCES.md`)

This document outlines the real-world research behind our data ingestion structures, our sample data strategy, and failure points in production.

---

## 1. SAP Procurement (Scope 1)

### A. Real-World Research
In SAP ERP, procurement items are recorded in standard database tables:
- `EKKO` (Purchasing Document Header)
- `EKPO` (Purchasing Document Item)
- `EKBE` (History per Document)
Exports from these tables use standard technical column names (German terms):
- `WERKS`: Plant (factory/facility identifier).
- `BUDAT`: Posting Date (when the procurement was officially entered).
- `MENGE`: Quantity of items.
- `MEINS`: Base Unit of Measure (e.g., L for liters, KG for kilograms).
- `DMBTR`: Amount in local currency.
- `MAKTX`: Material Description (text field containing details like "Diesel Fuel").

### B. Sample Data Design
Our sample dataset `sap_procurement_export.csv` tests:
- Columns in German technical headers.
- Date fields in both standard SAP `YYYYMMDD` and German European `DD.MM.YYYY` format.
- Plants mapped to regional grid intensities (`1000` -> Germany, `1100` -> Detroit US, `2000` -> Chennai India).
- Anomaly checks: Row with unknown plant `9999`, row with negative quantity `-150`, and row with unknown unit of measure `XYZ`.

### C. Production Breakdowns
*   **Column mapping changes**: If the SAP export is generated by a different report with English column names (`Plant`, `Date`, `Quantity`), the parser would break.
*   **Mitigation**: Production systems implement an "Ingestion Schema Mapper" UI where users drag and map their CSV columns to standard database attributes.

## 2. Utility Electricity (Scope 2)

### A. Real-World Research
Energy tracking portals (e.g. EnergyStar Portfolio Manager, Schneider Electric) log facility consumption. Utility bills report energy registers:
- `Current Reading` vs `Previous Reading`.
- `Multiplier`: Translates register differences into actual energy values (e.g. standard CT meters multiply by 10x or 100x).
- `Billing Period`: Utility cycles rarely align with calendar months (e.g., April 15 to May 14).

### B. Sample Data Design
Our sample dataset `utility_portal_export.csv` tests:
- Meter reading calculation check: `(Current - Previous) * Multiplier` must equal reported consumption.
- Date range splitting: A bill spanning April 15 to May 14 pro-rates 16 days to April and 14 days to May.
- Anomaly checks: Row where consumption math is incorrect (difference is 2000 kWh but reported is 5000 kWh), row with inverted dates, and row with unknown facility code `FAC-99`.

### C. Production Breakdowns
*   **Irregular bills**: If a facility misses a reading, a bill might span 90 days. Splitting this across 3 months could skew baseline comparisons.
*   **Mitigation**: Highlight irregular durations (> 45 days) to analysts for manual verification.

## 3. Corporate Travel (Scope 3)

### A. Real-World Research
Travel management systems like Concur or Navan expose REST endpoints (e.g., `/travel/bookings/` or `/v3.0/expense/entries`). They return structured JSON payloads:
- Flight segments with IATA airport codes (e.g. `JFK` for New York, `LHR` for London Heathrow) and cabin class (`Economy`, `Business`).
- Hotel stays reporting hotel name, country, and room-nights.
- Ground travel reporting distances in kilometers.

### B. Sample Data Design
Our simulated API endpoint returns:
- Short-haul flight (FRA to CDG, < 1600 km) vs long-haul flight (JFK to LHR).
- Hotel stays in Germany and India.
- Anomaly checks: Flight using a cabin class ("First") that falls back to Business Long-haul, and hotel stay reporting negative room-nights (`-2`).

### C. Production Breakdowns
*   **Unlisted Airport Codes**: If an employee books a flight between new airports not in our seed table (e.g., smaller regional fields), distance calculations fail.
*   **Mitigation**: We flag missing codes. In production, we would integrate a geocoding API (like Google Maps or Aviation Edge) to fetch airport coordinates dynamically.
